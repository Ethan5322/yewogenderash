## 3. Core Platform Rules

### 3.1 Trust First
The platform must feel safe, transparent, and auditable.

- Every campaign must have a verified owner profile.
- Every campaign must have a unique querycode and QR code.
- Every donation must be tied to exactly one campaign.
- Every payout must be recorded in a separate payout ledger.
- Every admin action must be logged.

### 3.2 No Fund Mixing
Funds for different campaigns must never be mixed.

- Each campaign gets its own donation ledger.
- Each campaign gets its own payout summary.
- Campaign owners may only see their own campaigns and balances.
- Only admin can see the full system-wide financial view.

The site must look polished, minimal, and trustworthy.

- Strong visual hierarchy.
- Consistent spacing.
- Professional color palette.
- No clutter.
- No exaggerated claims.

## 7. Verification and Trust Model

### 7.1 Biometric Verification

The system must support face capture for campaign owner identity verification.

- Capture live face.
- Perform liveness detection.
- Compare with uploaded ID and account identity.
- Mark status as pending, verified, rejected, or resubmit.

### 7.2 Campaign Author ID

Once verified, each campaign owner receives a public author profile with:

- Full name.
- Photo.
- Verification badge.
- Unique verification code.
- QR code.
- Mulesoo stamp.
- List of campaigns owned.
- Verification status and date.

### 7.3 Mulesoo Stamp

The Mulesoo stamp is a platform trust seal.

- It should appear only after admin approval.
- It must be visible on owner profile and verified campaigns.
- It must not be editable by the user.

## 8. Campaign and Querycode Rules

### 8.1 One Campaign, One Querycode

- Every campaign must have its own unique querycode.
- Every campaign must have its own QR code.
- A QR code must open only that campaign's quick-donate page.
- One querycode must never point to more than one campaign.

### 8.2 One Campaign, One Ledger

- Each campaign must have a separate donation ledger.
- Each campaign must have a separate payout summary.
- Campaign owners must not see other owners' campaigns or funds.
- Admin may see all ledgers.

### 8.3 Donation Routing
When a donor scans the QR:

- The system identifies the exact campaign.
- The donor lands on the campaign's quick-donate page.
- The donor sees campaign details and verified status.
- Payment is processed only for that campaign.

## 9. Donation Flow

### 9.1 Standard Donation Flow

1. User opens full campaign page.
2. User clicks donate.
3. User enters amount.
4. User selects payment method.
5. Backend creates a pending donation record.
6. Backend creates payment session with gateway.
7. User completes payment.
8. Gateway sends webhook.
9. Backend verifies webhook.
10. Donation status is updated.
11. Campaign total is updated.
12. Campaign owner receives WhatsApp alert.
13. Donor sees thank-you receipt.

### 9.2 Quick Donate Flow

1. User scans QR or enters querycode.
2. User lands on `/q/[queryCode]`.
3. User sees short campaign summary and payment box.
4. User chooses amount.
5. User pays.
6. Backend verifies payment.
7. Owner gets alert.
8. Thank-you screen appears.
9. Session closes or resets.

### 9.3 Donation State Machine

Donation status values must include:

- `PENDING`
- `SUCCESS`
- `FAILED`
- `CANCELLED`
- `REFUNDED`
- `DISPUTED`
